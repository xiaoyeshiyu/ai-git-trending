### ✨ simplex-chat/simplex-chat (12251★)

> **一句话**：这是一个不依赖用户标识符的私密聊天网络，提供 iOS、Android、桌面端和命令行客户端，让你在尽量少暴露元数据的前提下建立对话和群组连接。

- **它是什么**：SimpleX Chat 的核心不是“加密聊天”这么简单，而是尽量不让系统层面知道“你是谁”。README 明确强调它没有任何用户标识符，消息和连接都围绕私有链接、二维码和会话建立展开，同时支持移动端、桌面端和终端客户端。它还提供连接安全码验证、群组、机器人和目录等功能，已经不是单一聊天 App，而是一套完整的私密通信网络。

- **能解决什么痛点**：一类典型问题是传统 IM 必须绑定手机号、用户名或联系人列表，导致身份、社交关系和活跃时间容易被平台侧关联；SimpleX 试图把这类元数据暴露降到更低。另一类问题是团队或社区需要一个能跨设备使用、同时支持私密连接和群组交流的通信方案，而不是只依赖单个平台的封闭账号体系。

- **适合谁用**：重视隐私和元数据保护的个人用户，尤其是希望减少手机号、账号体系暴露的人。也适合做隐私通信、去中心化社交、聊天机器人和集成应用的开发者，README 里还专门提到了开发 bot、自动化和基于 SimpleX 平台的社交应用。

- **怎么上手**：README 给出的最直接方式是安装官方客户端，例如 iOS、Android、F-Droid、TestFlight 或 APK。最小可用流程是先装 App，再通过分享链接或扫码建立私密连接；项目页也提供了终端客户端入口，但 README 没有给出单行命令式的快速启动示例。

- **可以用在哪些场景**：适合做高隐私要求的个人或小团队通信，比如不想把手机号和社交关系绑定到聊天工具里。也适合建立面向特定社群的私有群组，例如安全研究、隐私社区或跨平台协作群。对于开发者，还可以拿它做聊天机器人、通知自动化或与其他应用联动的入口。

- **技术看点**：项目用 Haskell 开发，说明核心协议和服务端逻辑偏重类型安全与可维护性。README 里写得很明确，它采用双重 ratchet 端到端加密，并在此基础上再加一层加密，同时围绕“无用户标识符”的连接模型设计，这是它和常规 IM 最大的技术差异。

- **近期动向与发展方向**：最近提交很集中，基本围绕 6.5.4 到 6.5.6 的版本迭代在推进，说明项目仍处于持续发布节奏。近期工作重点主要是核心逻辑修复、链接处理、群组权限、跨平台构建和桌面/移动端体验修正，例如修复管理员降权后的群链接问题、阻止不合规的 obfuscated simplex links、iOS 里直接打开聊天消息中的连接链接、桌面端下载器行为修正。整体看，项目在稳定打磨现有能力，同时继续补齐隐私连接和群组机制的边角行为。

- **同类对比**：README 没有直接点名竞品，但它明显是在和传统“手机号即身份”的聊天软件做区分，强调自己的无标识符设计和元数据保护。文中引用了 Privacy Guides、Whonix 等外部推荐，也是在强化它在隐私通信领域的定位。

- **注意事项**：项目成熟度不低，创建于 2019 年，星标已经过万，更新也很频繁，但 open issues 有 1139 个，说明活跃度高的同时，待处理问题也不少。对新用户来说，上手门槛会高于普通 IM，因为它的连接方式、身份模型和安全验证方式都不属于日常聊天软件的默认思路；另外 README 很长，功能和安装入口很多，但“快速上手”路径并不算简洁。版本迭代频繁，跨端功能又多，升级时要留意兼容性和行为变化。

### ✨ xbtlin/ai-berkshire (1464★)

> **一句话**：把巴菲特、芒格、段永平、李录的投资方法拆成 Claude Code 可调用的研究流程，让一个人用多 Agent 跑出结构化投研报告。

- **它是什么**：AI Berkshire 是一套面向 Claude Code 的投资研究 Skill 集合，核心是把价值投资研究拆成固定流程和命令入口，比如 `/investment-research`、`/investment-team`、`/industry-funnel`、`/portfolio-review` 等。它不是简单让 AI 写一篇股票分析，而是要求多视角并行研究、交叉验证财务数据、输出明确结论、价格区间和风险判断。README 中还展示了实盘收益记录、四大师视角对抗、信息置信度评级、快速否决清单等机制。

- **能解决什么痛点**：直接问 LLM 分析股票时，常见问题是结论含糊、格式不稳定、数据计算容易出错，最后很难用于真实决策。这个项目把投研流程固定成可复用 Skill，并通过多 Agent、反向检验、财务数据校验等方式，减少“看起来很完整但无法落地”的 AI 分析。

- **适合谁用**：适合已经在使用 Claude Code、希望把 AI 纳入投资研究流程的个人投资者或研究员。也适合想搭建内部投研模板库、需要批量分析公司、行业、财报和持仓组合的金融内容创作者或投研团队。

- **怎么上手**：`git clone https://github.com/xbtlin/ai-berkshire.git && cp ai-berkshire/skills/*.md ~/.claude/commands/`，然后在 Claude Code 中调用 `/investment-team 美团` 或 `/investment-research 腾讯`。

- **可以用在哪些场景**：
  1. 对单家公司做深度研究，例如从商业模式、护城河、管理层、估值和风险角度分析腾讯、美团、拼多多。
  2. 做行业漏斗筛选，例如从 AI 算力、核电、白酒、半导体产业链中筛出少数值得继续研究的标的。
  3. 管理已有投资组合，例如用 `/portfolio-review` 检查仓位集中度、再平衡逻辑和投资论文是否被证伪。

- **技术看点**：项目的核心设计不是模型微调，而是基于 Claude Code commands 的流程工程：用 16 个 Skill 固化投研入口，再用多 Agent 并行模拟不同研究角色。财务严谨性方面，README 明确提到使用 Python `decimal.Decimal` 做精确十进制计算，并要求关键数据至少两个独立来源交叉验证。

- **近期动向与发展方向**：最近提交非常活跃，6 月 25 日一天内有大量报告新增、目录整理和“瓶颈猎手”扫描日报更新，说明项目当前重点在持续产出投资研究报告和行业/个股跟踪内容。提交中多次出现 Claude 生成的扫描记录，也能看出作者正在把该框架用于高频市场观察；同时有 PR 合并和报告迁移，社区已有少量外部贡献，但核心维护仍明显集中在主作者及 Claude 辅助流程上。

- **同类对比**：README 明确对比的是“直接问 Claude 分析股票”的方式。AI Berkshire 的差异在于强制输出明确投资结论、固定研究结构、引入四大师视角对抗、多 Agent 并行和财务校验，而不是每次生成一篇风格不稳定的通用分析。

- **注意事项**：项目创建时间较新，但更新频率很高，Stars 增长较快；目前有 11 个 open issues、5 位贡献者，仍处于快速演进阶段。它依赖 Claude Code 的使用环境，且研究框架本身不等于投资建议，README 虽展示实盘收益，但历史收益不能代表未来表现。由于近期提交中包含大量报告和扫描内容，仓库可能兼具“工具框架”和“投研内容库”两种属性，使用前最好先区分自己需要的是 Skill 模板还是具体研究报告。

- **GitHub**：[xbtlin/ai-berkshire](https://github.com/xbtlin/ai-berkshire)

### ✨ every-app/open-seo (2615★)

> **一句话**：把关键词研究、排名追踪、外链分析、站点审计和 AI 搜索可见性放进可自托管的 SEO 工作台，并能通过 MCP 让 Codex、Claude Code 等 AI Agent 直接调用 SEO 数据。

- **它是什么**：OpenSEO 是一个开源 SEO 平台，定位为 Semrush 和 Ahrefs 的替代方案，核心功能包括关键词研究、排名追踪、域名洞察、外链分析、站点审计、AI 品牌可见性和 AI 搜索提示探索。它支持 Docker 本地部署和 Cloudflare 自托管，也提供托管版 `openseo.so`。项目还内置 MCP Server 和 Agent Skills，让 AI Agent 可以在编辑器或聊天环境里查询关键词、SERP、外链和域名数据。
- **能解决什么痛点**：一是传统 SEO 套件订阅费高、功能臃肿，小团队或独立开发者很难只为少量关键词和站点审计持续付费。二是 SEO 数据通常和开发工作流割裂，OpenSEO 通过 MCP 把 DataForSEO、排名、外链等数据接入 Codex / Claude Code，方便在写内容、改页面或做竞品分析时直接调用。
- **适合谁用**：适合想自托管 SEO 系统的独立站开发者、增长工程师和内容团队；也适合正在把 Codex、Claude Code 等 Agent 接入日常工作流，希望让 AI 直接读取 SEO 数据的技术团队。
- **怎么上手**：Docker 最小启动方式：`cp .env.example .env && docker compose up -d`，之后在 `.env` 中配置 `DATAFORSEO_API_KEY` 并访问默认端口 `3001`。
- **可以用在哪些场景**：跟踪独立站核心关键词在桌面端和移动端的排名变化；为博客或产品页做关键词研究并筛选下一批内容主题；审计站点技术问题、查看外链来源，并分析品牌在 ChatGPT / Google AI Overview 等 AI 搜索结果中的露出情况。
- **技术看点**：项目用 TypeScript 构建，支持 Docker 与 Cloudflare 两条自托管路径，数据层依赖 DataForSEO 的按量 API。MCP Server 和预置 Agent Skills 是它区别于传统 SEO 面板的关键设计，能把 SEO 查询变成 AI Agent 可调用的工作流能力。
- **近期动向与发展方向**：最近提交非常密集，6 月中下旬连续发布 v0.0.21、修复 onboarding 溢出和邮箱验证、改进 onboarding chat、迁移到 DigitalOcean、优化 MCP 输出校验，并加入 keyword research、DataForSEO task queue、计费状态同步等内容。整体看项目处于快速迭代期，重点从基础 SEO 功能扩展到 Agent 工作流、托管版商业化、计费和稳定性修复；但当前贡献者数量为 1，开发节奏主要依赖单一维护者。
- **同类对比**：README 明确对标 Semrush 和 Ahrefs，差异在于 OpenSEO 可自托管、按 DataForSEO API 用量付费，并提供 MCP / Agent Skills；Semrush 和 Ahrefs 更成熟、数据体系更完整，但订阅成本和封闭性更高。
- **注意事项**：项目创建于 2026-02-27，仍很年轻，虽然 Stars 增长快且近期更新频繁，但成熟度还需要观察。自托管并不等于完全免费，关键词、外链、AI 搜索等数据仍依赖 DataForSEO 付费 API，部分外链能力还有额外门槛。Docker 默认是本地单用户无认证模式，若要公网访问，README 建议使用 Cloudflare 自托管并仔细阅读部署文档。
