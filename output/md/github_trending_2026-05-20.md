### ✨ rtk-ai/rtk (51148★)

> **一句话**：它会悄悄拦下 AI 助手准备执行的 `git status`、`pytest`、`docker logs` 这类命令，把原本几百上千行的终端输出压缩成更短但还能看懂的结果，再喂给大模型，直接少吃掉 60% 到 90% 的 token。

- **它是什么**：rtk 是一个用 Rust 写的命令行代理层，核心思路不是替代 Git、测试工具或容器命令，而是在这些命令执行后，对输出做过滤、分组、截断和去重，再把更紧凑的结果交给 LLM。它支持 100 多种常见开发命令，覆盖文件读取、Git、测试、Lint、Docker、kubectl、AWS CLI 等场景，还能通过 hook 自动把 AI 助手发出的 Bash 命令改写成 `rtk ...`。
- **能解决什么痛点**：
  - 让 Claude Code、Copilot、Gemini CLI 这类编程助手少读无意义输出，比如 `git diff` 的大段上下文、测试通过日志、重复报错、进度条和 ASCII 装饰，避免上下文窗口被噪声挤满。
  - 在多轮调试里，开发者经常要反复跑 `cargo test`、`pytest`、`docker logs`、`grep`，原始输出很容易把 token 成本和响应延迟一起拉高，rtk 就是专门压这部分开销的。
- **适合谁用**：
  - 重度依赖 Claude Code、Codex、Gemini CLI、Cursor、Windsurf、Cline 等 AI 编程代理的开发者。
  - 经常在终端里做测试、排错、Git 操作、容器排障的后端工程师、Rust/Go/Python 开发者和运维 SRE。
- **怎么上手**：`brew install rtk && rtk init -g`
- **可以用在哪些场景**：
  - 用 Claude Code 或 Codex 在大型仓库里做日常改代码时，把 `git status`、`git diff`、`rg`、`cat` 的输出压缩后再送进模型，减少上下文浪费。
  - 跑 `cargo test`、`pytest`、`go test`、`ruff check` 这类命令时，只把失败信息和关键摘要交给 AI，让它更快定位错误而不是先“读日志”。
  - 看 `docker logs`、`kubectl logs`、AWS CLI 返回结果时先做去重和裁剪，适合线上排障或云资源巡检时配合 AI 助手一起分析。
- **技术看点**：项目采用单个 Rust 二进制、零依赖分发，安装和部署都比较轻，README 声称额外开销低于 10ms。设计上不是做通用压缩，而是按命令类型做有针对性的输出重写，比如 Git、测试框架、lint、云 CLI 分别使用不同规则，这比简单截断更适合给 LLM 喂上下文。
- **同类对比**：README 没有明确列出竞品，但它的差异点很明确：不是新的 AI IDE，也不是模型网关，而是夹在“AI 助手调用 shell”这一层专门做终端输出压缩；同时支持 hook 自动改写命令，不要求用户每次手动把原命令改成 `rtk ...`。
- **注意事项**：项目创建于 2026-01-22，但 Star 增长极快，说明关注度非常高；同时 Open Issues 达到 958，表明生态扩张快、问题反馈也很多，稳定性和兼容性仍值得观察。README 很详细，安装、支持命令、不同代理接入方式都写得比较清楚；不过自动改写目前明确依赖 Bash tool calls，像 Claude Code 内建的 `Read`、`Grep`、`Glob` 并不会自动走 hook，这类场景需要手动改用 shell 命令或直接调用 `rtk read`、`rtk grep`。另外项目较新，命令支持面又很广，接入到团队工作流前建议先在自己的代理和常用命令集上做一轮兼容性验证。

- **GitHub**：[rtk-ai/rtk](https://github.com/rtk-ai/rtk)

### ✨ pascalorg/editor (15953★)

> **一句话**：它把“画楼层、拉墙体、开门窗、摆家具、切换楼层视图”这些建筑编辑动作直接搬进浏览器里，做成了基于 React Three Fiber 和 WebGPU 的 3D 建筑编辑器。

- **它是什么**：这是一个面向建筑/室内场景的在线 3D 编辑器，仓库采用 Turborepo monorepo 组织，拆成 `core`、`viewer` 和 `apps/editor` 三层。`core` 负责场景节点、状态管理、几何系统和空间查询，`viewer` 负责 Three.js 渲染与默认相机控制，`editor` 再叠加墙体绘制、区域创建、物件放置、选择管理等交互能力。
  README 里给出的节点模型很完整，支持 Site、Building、Level、Wall、Slab、Roof、Zone、Scan、Guide、Item 等建筑实体，并且通过系统在渲染循环中增量更新几何。

- **能解决什么痛点**：一是做浏览器内建筑编辑时，开发者通常要自己拼“场景数据结构 + 3D 渲染 + 几何生成 + 选择/工具交互 + 撤销重做”，这个项目把这些基础设施打包好了。二是墙体、楼板、门窗开孔、楼层显示模式、物件放置校验这类建筑编辑里的细节逻辑很难从零写，仓库已经内置了对应 system 和空间校验能力。

- **适合谁用**：适合想做 Web 端 BIM/室内设计/空间规划产品的前端团队，尤其是技术栈本来就用 React、Three.js、Next.js 的团队。也适合需要二次开发 3D 建筑编辑能力的创业项目或内部数字化平台团队。

- **怎么上手**：`bun install && bun dev`

- **可以用在哪些场景**：
  1. 做在线户型编辑器，让用户在浏览器里画墙、分区、摆放门窗和家具。
  2. 做建筑/楼层可视化后台，把站点、建筑、楼层、区域按层级组织并切换 exploded/solo 视图。
  3. 做扫描数据辅助建模工具，把 3D scan 和 guide image 作为参考底图叠加到编辑流程里。

- **技术看点**：项目把“数据节点”和“Three.js 对象”解耦：场景数据存在 Zustand store 里，渲染对象通过 scene registry 建立映射，几何更新则交给各类 system 按 dirty nodes 增量处理，这种架构比把业务逻辑直接塞进组件里更适合复杂编辑器。
  另外它使用 React 19、Next.js 16、React Three Fiber、WebGPU renderer、Zundo 撤销重做、three-bvh-csg 布尔几何，说明它不是简单展示场景，而是认真在做可编辑的建筑几何系统。

- **同类对比**：暂无明显同类对标。

- **注意事项**：项目创建时间很新（2025-10），但星标增长很快，说明关注度高；同时当前有 46 个 open issues、21 位贡献者、最近仍在更新，活跃度不错，但整体仍偏早期，接口和架构还有继续演进的可能。README 对架构讲得比较细，适合二次开发者理解内部设计，不过真正上手需要同时理解 React Three Fiber、Three.js、Zustand、几何系统和 monorepo 协作，门槛不低。另一个实际限制是它依赖较新的前端栈和 WebGPU 渲染路径，落地前需要评估浏览器兼容性与团队技术储备。

- **GitHub**：[pascalorg/editor](https://github.com/pascalorg/editor)

### ✨ frappe/erpnext (34365★)

> **一句话**：把记账、库存、采购、销售、生产、资产和项目管理放进同一套系统里，让企业日常经营不再靠一堆彼此割裂的软件和表格拼起来。

- **它是什么**：ERPNext 是一套开源 ERP 系统，覆盖财务、订单与库存、制造、资产、项目等核心业务模块。它不只是做“进销存”或“财务记账”，而是把企业从下单、采购、入库、生产到开票、收款、项目交付这些流程串成一体。底层基于 Frappe Framework，提供完整的 Web 应用能力、认证和 API。
- **能解决什么痛点**：一是很多中小企业会同时用表格、财务软件、仓库系统和项目工具，数据分散、口径不一致，月底对账和库存核对非常痛苦；ERPNext 把这些数据放进同一业务系统里。二是制造和贸易场景里，订单履约、备料、出货、开票常常跨多个角色协作，手工流转容易漏单、超卖或账货不一致。
- **适合谁用**：需要自建或二次开发业务系统的中小企业技术团队；为客户做数字化交付的实施服务商、ERP 顾问和 Python/Frappe 开发者。
- **怎么上手**：`docker compose -f pwd.yml up -d`
- **可以用在哪些场景**：制造企业管理 BOM、工单、产能和物料消耗；贸易或零售团队统一处理采购、库存、销售订单、发货和应收应付；服务型公司按项目跟踪任务、工时、问题和利润情况。
- **技术看点**：项目建立在 Frappe Framework 之上，后端以 Python 为主，具备数据库抽象、认证和 REST API，适合做业务定制。前端侧配合 Vue 技术栈的 Frappe UI，说明它不是单纯的老式后台，而是在往现代 Web 应用体验上演进。
- **同类对比**：README 未明确点名竞品，但从定位上看，它强调“100% 开源”与“覆盖完整企业流程”，区别于只做单点功能的软件组合，也区别于闭源商业 ERP 的高授权成本模式。
- **注意事项**：这是 2011 年创建、长期维护的大型成熟项目，贡献者 916、Stars 3.4 万，活跃度和生态都很强；但同时 Open Issues 达 2079，说明系统复杂度高、边界广，上手和实施都不是“装完就用”那么简单。README 提供了 Docker 快速试用和本地 bench 安装路径，文档与社区资源比较完整；不过快速 Docker 方案明确标注为一次性演示环境，不适合直接用于生产，正式部署还需要结合 Frappe Docker、升级维护和自定义应用方案来评估。

- **GitHub**：[frappe/erpnext](https://github.com/frappe/erpnext)

### ✨ HKUDS/ViMax (5586★)

> **一句话**：把一句创意、一个剧本、一本小说甚至一张自拍，自动拆成脚本、分镜、角色设定和多镜头视频，直接产出带叙事结构的 AI 视频。

- **它是什么**：ViMax 是一个面向长流程视频生成的多智能体框架，定位不是单纯“生几秒视频”，而是把导演、编剧、制片和视频生成串成一条自动化流水线。它支持从 Idea2Video、Novel2Video、Script2Video 到 AutoCameo 等模式，覆盖创意扩写、长脚本拆分、分镜设计、角色一致性控制和最终视频生成。
- **能解决什么痛点**：一是长视频生成里最麻烦的“跨镜头一致性”问题，比如人物长相、场景风格、镜头衔接经常前后跑偏；二是从故事到视频中间要人工补的环节太多，像脚本整理、分镜规划、参考图管理、镜头设计，普通创作者很难一次性串起来。
- **适合谁用**：适合在做 AI 视频工作流、内容生成平台或多智能体编排系统的 Python 开发者；也适合需要把小说、短篇故事、宣传创意快速转成视频样片的 AI 创作者和内容团队。
- **怎么上手**：文档未提供快速上手示例。
- **可以用在哪些场景**：把网文或短篇小说自动改编成分集短剧视频；给品牌营销团队快速生成概念广告片或剧情式宣传片样片；做“用户上传照片即客串剧情视频”的互动内容产品。
- **技术看点**：项目核心是多智能体视频生成架构，不只调一次模型，而是把长脚本生成、分场景拆解、分镜设计和一致性控制组织成完整链路。README 还提到基于 RAG 的长文本脚本设计能力，说明它更关注长叙事输入到多场景输出的结构化转换，而不是单镜头生成。
- **同类对比**：README 明确对标的是当前只能生成短片段、缺少剧情结构和一致性控制的 AI 视频工具；ViMax 的差异点在于强调“端到端叙事生产”，把脚本、分镜、音画绑定和角色连续性一起纳入流程。
- **注意事项**：项目创建于 2025-03-30，星标增长很快，但仍偏早期，10 位贡献者、33 个开放 issue，说明活跃但还在快速演进阶段。README 展示效果和定位很清楚，不过快速安装、最小运行示例和具体依赖说明在给定材料里不完整；另外项目更新时间到 2026-05-20，变更节奏可能较快，接入时要留意接口和工作流调整风险。
