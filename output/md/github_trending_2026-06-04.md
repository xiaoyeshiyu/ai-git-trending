### ✨ aquasecurity/trivy (35436★)

> **一句话**：把容器镜像、Kubernetes 集群、代码仓库、文件系统和云环境一起扫一遍，直接找出漏洞、配置错误、泄露的密钥以及 SBOM 信息。

- **它是什么**：Trivy 是 Aqua Security 维护的综合安全扫描器，覆盖容器镜像、文件系统、远程 Git 仓库、虚拟机镜像和 Kubernetes 等常见交付对象。它不只查 CVE，还能扫描 IaC 配置错误、代码或配置中的 secrets、软件许可证，以及生成/利用 SBOM 来识别依赖与风险。对 DevSecOps 流程来说，它相当于把“漏洞扫描 + 配置审计 + 密钥泄露检查”合到了一套命令里。
- **能解决什么痛点**：一是很多团队会在 Docker 镜像、Terraform/Helm/K8s 配置、源码仓库里分别接入不同安全工具，规则分散、结果格式不统一，Trivy 能集中完成这些检查。二是发布前常会漏掉基础镜像里的系统包漏洞、仓库里误提交的密钥或 IaC 配置错误，等上线后才暴露风险，Trivy 适合前置到 CI/CD 里做拦截。
- **适合谁用**：容器化应用团队、平台工程师、DevOps/SRE、安全工程师；以及需要在 CI 流水线里统一做镜像、仓库、Kubernetes 安全检查的团队。
- **怎么上手**：`trivy image python:3.4-alpine`
- **可以用在哪些场景**：在构建 Docker 镜像后立即扫描基础镜像和依赖包漏洞，决定是否允许发布；在扫描 Terraform、Ansible、Kubernetes YAML 时提前发现高风险配置；在代码仓库或本地目录里检查误提交的密码、私钥、Maven settings 凭据等敏感信息。
- **技术看点**：项目把“扫描器”和“扫描目标”做了清晰拆分：同一套框架可以面向 image、fs、repo、k8s 等目标，复用漏洞、misconfig、secret、license 等能力。Go 实现、分发渠道多，既可本地二进制运行，也可直接用 Docker 镜像或集成到 GitHub Actions、Kubernetes Operator、VS Code 插件中，落地门槛比较低。
- **近期动向与发展方向**：最近非常活跃，6 月 1 日刚发布 v0.71.0，近几天连续有功能、修复和 CI 改进提交。开发重点很明确：一是在增强扫描准确性和稳定性，比如镜像中嵌入 SBOM 的 OS 包去重、SPDX nil root component 防护、Terraform plan 嵌套值渲染修复；二是在扩展检测能力，比如新增 Maven rules 检测 settings.xml / settings-security.xml 中的密码与 passphrase；三是在持续做工程治理，包括依赖升级、最小权限 GitHub App token、测试离线扫描、Helm Chart 同步升级。贡献者分布也比较活跃，既有核心开发者持续推进，也有 bot 和外部贡献者参与维护。
- **同类对比**：README 未直接点名竞品，但 Trivy 的差异点很明显：它不是只做镜像漏洞扫描，而是把漏洞、SBOM、IaC misconfig、secret、license 多种检查统一到一个入口里，并且扫描目标覆盖容器、仓库、文件系统和 Kubernetes，适合想减少工具栈碎片化的团队。
- **注意事项**：项目成熟度高，2019 年开始维护、Star 很高、贡献者超过 500，且最近仍保持高频更新，适合生产使用；但 open issues 有 245 个，说明功能面广、边界情况也多，复杂场景下仍需要结合自身环境验证规则和误报情况。README 和文档入口完整，安装方式也多，不过由于能力覆盖面很广，初次接入时需要先明确扫描目标和 scanners 组合，否则容易把结果噪音拉高。另有 canary 构建可用，但官方明确提示不建议直接用于生产。

### ✨ opendataloader-project/opendataloader-pdf (23350★)

> **一句话**：把普通 PDF 直接拆成适合 RAG/LLM 消费的 Markdown、带坐标的 JSON、HTML，或者进一步自动补结构生成可读屏的 Tagged PDF。

- **它是什么**：这是一个面向两类需求的 PDF 处理项目：一类是把 PDF 提取成 AI 可直接使用的结构化数据，另一类是把原本没做无障碍标记的 PDF 自动补成 Tagged PDF。它支持本地确定性解析，也支持混合 AI 模式处理复杂表格、扫描件、公式和图片描述，并提供 Python、Node.js、Java 三套接入方式。README 里还明确把“每个元素都有 bounding box”“阅读顺序恢复”“自动标签化”作为核心卖点。

- **能解决什么痛点**：
  1. 做知识库/RAG 时，普通 PDF 往往会出现段落顺序错乱、表格碎裂、标题层级丢失，导致切块和引用效果很差；这个项目能输出带坐标和语义类型的结构化结果。
  2. 企业或机构面对 PDF 无障碍合规时，人工修复一份文档成本高且难规模化；它提供开源的自动标签化流程，可先把未标记 PDF 转成 Tagged PDF，作为后续 PDF/UA 流程基础。

- **适合谁用**：
  1. 做 RAG、文档理解、企业知识库接入的 Python/Java 工程师。
  2. 需要批量处理历史 PDF、推进无障碍整改的文档平台团队、政企信息化团队或内容处理团队。

- **怎么上手**：`pip install -U opendataloader-pdf`
  最小调用示例：`opendataloader_pdf.convert(input_path=["file1.pdf"], output_dir="output/", format="markdown,json")`

- **可以用在哪些场景**：
  1. 把论文、财报、合同批量转成 Markdown/JSON，接入内部 RAG 检索和问答系统。
  2. 处理扫描版档案、票据或历史文档，结合 OCR 产出可检索、可引用的结构化内容。
  3. 给政府、教育、金融等需要合规交付的 PDF 批量补标签，先生成 Tagged PDF 再进入无障碍审校流程。

- **技术看点**：README 给出的差异点比较明确：本地模式强调确定性输出和速度，混合模式把复杂页面路由到 AI 后端，兼顾精度和成本；同时对每个元素保留 bounding box，便于做引用定位、证据高亮和二次加工。另一个很强的点是无障碍方向，项目强调依据 Well-Tagged PDF 规范，并与 PDF Association、veraPDF 相关团队协作，这对需要做合规验证的团队有参考价值。

- **近期动向与发展方向**：最近提交非常活跃，5 月到 6 月连续更新，重点明显集中在三条线上：一是 hybrid 模式能力增强，比如保留原始响应、对象 ID、整页渲染和证据叠加相关元数据；二是 JSON/HTML 输出继续补细节，如 `ai_score`、`pdfua_tag`、图片 alt 字段统一、HTML 字号和布局属性补充；三是无障碍自动标签化持续打磨，连续修复 PDF/UA 相关表格、注释、结构元素 ID、日志与校验问题。整体看不是“只修小 bug”，而是在朝“更可审计的 AI 提取 + 更规范的无障碍输出”两端同步推进，且最近贡献者不只 1 人，社区协作是活跃的。

- **同类对比**：README 明确给了 benchmark，对标了 docling、marker、unstructured、mineru、pymupdf4llm、markitdown 等项目。它主打的差异不是单纯“能解析 PDF”，而是“混合模式精度排前、表格表现强、每元素带坐标、支持自动标签化到 Tagged PDF”。如果你的需求只是轻量文本抽取，未必需要它这套较完整流程；但如果你同时在意阅读顺序、复杂表格、扫描件和无障碍输出，它的定位更完整。

- **注意事项**：项目很新，创建于 2025-05-13，但星标增长很快，说明关注度高、传播强；同时 open issues 有 57 个，意味着功能迭代快，边界情况还在持续收敛。运行上要求 Java 11+，Python 方案本质上也会拉起 JVM，批量调用方式需要按 README 建议一次性处理，避免频繁启动进程。混合模式、OCR、公式和图片描述虽然能力更强，但部署链路也更复杂；另外 README 说明 PDF/UA 导出属于企业增强项，开源部分主要覆盖 auto-tagging 到 Tagged PDF，不要把它和完整 PDF/UA 商业交付能力混为一谈。

- **GitHub**：[opendataloader-project/opendataloader-pdf](https://github.com/opendataloader-project/opendataloader-pdf)

### ✨ lyogavin/airllm (18956★)

> **一句话**：它把超大语言模型的权重按层拆开、边加载边计算，让原本要几十 GB 显存才能跑的 70B/405B 模型，能在 4GB/8GB 显存设备上完成推理。

- **它是什么**：AirLLM 是一个面向大模型推理的内存优化方案，重点不是训练，而是把超大模型“塞进”低显存设备里跑起来。README 给出的核心能力很明确：无需蒸馏、剪枝，70B 模型可在单张 4GB GPU 上推理，甚至支持在 8GB 显存上运行 Llama 3.1 405B。它还提供 `AutoModel.from_pretrained(...)` 这样的接口，尽量让使用方式接近 Hugging Face/Transformers 的 привыч惯写法。
- **能解决什么痛点**：
  - 手上只有消费级小显存显卡，想本地验证 70B 级模型效果时，传统加载方式根本放不进显存。
  - 下载了超大模型后，常常卡在“显存不够”或“加载太慢”，AirLLM 通过分层加载、预取和可选压缩，缓解这类推理落地问题。
- **适合谁用**：
  - 想在本地低配 GPU 或 Apple Silicon 设备上试跑开源大模型的 LLM 开发者、研究者。
  - 做私有化验证、离线推理 PoC、模型兼容性测试的算法工程师或平台工程师。
- **怎么上手**：最简安装是 `pip install airllm`；最小用法是通过 `from airllm import AutoModel` 后执行 `AutoModel.from_pretrained("garage-bAInd/Platypus2-70B-instruct")` 加载模型并调用 `generate(...)` 推理。
- **可以用在哪些场景**：
  - 在只有 4GB/8GB 显存的测试机上，先验证 70B/405B 模型的回答质量，再决定是否投入更高算力部署。
  - 企业内部做敏感数据问答 PoC，希望先本地离线跑大模型，而不是直接上云推理服务。
  - 教学或研究场景里演示超大模型推理机制，尤其适合讲解“分层加载 + 磁盘换显存”的工程取舍。
- **技术看点**：项目核心思路是把原始模型先按层拆分保存，推理时按需加载，从“显存瓶颈”转成“磁盘 I/O + 调度”问题。README 还提到预取（prefetching）和基于 block-wise quantization 的压缩，可在几乎不明显损失精度的前提下把推理速度再提到最高约 3 倍。
- **近期动向与发展方向**：从最近提交看，2024 年下半年有实质功能更新，包括支持 Qwen2.5、修复 MLX 模型加载、加入 CPU inference 与非分片模型支持；而 2025 之后提交明显变少，主要是 README 和项目信息维护，2026 年新增 `funding.json`。这说明项目近阶段更偏向维护和兼容性修补，而不是高频新增核心能力；社区有少量外部贡献者参与修文档和修兼容问题，但整体开发活跃度较 2024 年有所下降。
- **同类对比**：README 没有直接点名竞品，但它的差异点很明确：不是主打传统量化部署框架，而是优先解决“超大模型在极低显存设备上的可运行性”，并强调在不依赖蒸馏、剪枝的情况下完成推理。与常见只强调量化降显存的方案相比，它更强调分层加载和磁盘换内存的工程路径。
- **注意事项**：项目热度很高，但当前有 159 个 open issues，说明真实使用中兼容性和环境问题并不少，尤其涉及 Hugging Face 模型、磁盘缓存、MacOS/MLX、Jupyter 环境时要有排障准备。README 文档算比较完整，给了 Quickstart、FAQ、Colab 和多模型示例，但也明确提示分层拆模非常占磁盘空间；此外最近功能开发节奏放缓，如果你依赖新模型第一时间支持或稳定企业级维护，需要额外评估风险。

- **GitHub**：[lyogavin/airllm](https://github.com/lyogavin/airllm)
