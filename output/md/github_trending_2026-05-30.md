### ✨ run-llama/liteparse (7427★)

> **一句话**：把 PDF、Office 文档和图片在本地直接拆成带坐标的文本、JSON 或截图，方便你喂给 LLM、做检索或跑批处理。

- **它是什么**：LiteParse 是一个以 Rust 为核心的本地文档解析器，重点不是“花哨 AI”，而是把文档里的文字和版面位置快速、稳定地提取出来。它能解析 PDF，也能先把 DOCX/XLSX/PPTX、图片转成 PDF 再处理，输出结构化 JSON、纯文本和页面截图。项目还内置 Tesseract，并支持接入 HTTP OCR 服务，所以既能零配置跑起来，也能按精度需求替换 OCR 后端。
- **能解决什么痛点**：
  - 做 RAG、知识库清洗时，普通文本提取往往丢掉版面结构，表格、多栏、页内定位很难还原；LiteParse 会保留 bounding box，方便后续切块、定位和高亮。
  - 本地或内网环境里，很多团队不方便把合同、财报、内部文档上传到云端；LiteParse 支持离线运行，不依赖云服务。
- **适合谁用**：
  - 做文档清洗、检索增强生成（RAG）、信息抽取的 Python / Node.js 工程师。
  - 需要在本地批量处理 PDF、扫描件、Office 文档的后端工程师或数据工程师。
- **怎么上手**：`pip install liteparse && lit parse document.pdf`
- **可以用在哪些场景**：
  - 把公司内部 PDF 手册、产品文档批量转成结构化文本，接入向量库或搜索系统。
  - 解析发票、报告、合同扫描件，结合 OCR 和坐标信息做字段抽取或人工复核。
  - 给 LLM Agent 生成文档页面截图，让模型不仅看文本，还能看图表、布局和页面上下文。
- **技术看点**：核心用 Rust 实现，底层走 PDFium 做空间文本提取，再把原生文本和 OCR 结果合并，并通过网格投影重建版面。它同时提供 Rust、Python、Node.js、WASM 多端绑定，CLI 也统一，适合嵌入不同技术栈。
- **同类对比**：README 明确把它和 LlamaParse 区分开：LiteParse 主打本地、轻量、开源、无专有 LLM 依赖；而更复杂的表格、多栏、手写体、扫描 PDF 等生产级难题，官方建议直接用云端的 LlamaParse。
- **注意事项**：项目创建于 2026-02-09，时间很新，但到 2026-05-30 已有 7427 Stars、15 个 open issues、19 位贡献者，热度和迭代速度都很高。README 文档比较完整，安装、CLI、OCR、自定义 HTTP OCR API 都写得清楚；不过仓库还特别标注了 “LiteParse V1” 旧代码入口，说明近期已有版本演进，接口和行为后续仍可能继续调整。另一个现实限制是：非 PDF 输入依赖 LibreOffice / ImageMagick，复杂文档效果也取决于 OCR 引擎质量，不适合把它当成“所有文档都能完美还原”的全能解析器。

- **GitHub**：[run-llama/liteparse](https://github.com/run-llama/liteparse)

> **一句话**：把“采集环境数据 → 训练世界模型 → 用 MPC 规划评测”这条研究流水线收成一套统一接口，方便你在标准环境里复现实验、跑基线、测泛化。

- **能解决什么痛点**：一是 world model 实验常常被“环境接入、数据格式转换、评测脚本不统一”拖慢，这个项目把这些公共部分提前封装好了。二是做泛化评测时，很多人要自己造 distribution shift 测试集；这里已经内置 factors of variation，可直接测试光照、纹理、动力学等变化下的 zero-shot generalization。

- **技术看点**：项目把数据层做成了小型 format registry，支持 lance、hdf5、folder、video、LeRobot 等多种格式，读写、转换和自动识别都统一在同一套接口里。README 还给了吞吐与存储基准，LanceDB 在本地和 S3 场景下明显优于 HDF5，这对大规模轨迹数据训练很有参考价值。

### ✨ Biohub/esm (2580★)

> **一句话**：把一条蛋白质序列直接送进大模型，就能拿到表示向量、结构预测，甚至进一步做蛋白功能理解与分子设计。

- **它是什么**：这是 Biohub 开源的蛋白质基础模型项目，核心包含三部分：蛋白语言模型 ESMC、结构预测模型 ESMFold2，以及基于 68 亿蛋白构建的 ESM Atlas。它不是单纯做“序列分类”，而是试图把蛋白从序列、结构到功能关系放进同一套“world model”里，用于预测、检索、解释和设计。
  README 里已经给出了两条主线路：一是本地通过 Hugging Face 跑模型，二是通过 Biohub 平台 API 做推理；另外还开放了 SAE（稀疏自编码器）来解释模型内部学到的生物学特征。

- **能解决什么痛点**：
  1. 做蛋白研究时，很多任务过去要依赖 MSA、结构模板或多套分散工具；这里可以直接从单条氨基酸序列出发拿到 embedding 或结构结果，降低前处理门槛。
  2. 做蛋白互作/抗体设计时，传统结构预测和候选筛选流程慢、试错成本高；README 明确强调 ESMFold2 支持单序列模式加速折叠，并已用于实验验证的 binder 设计流程。

- **适合谁用**：
  1. 做计算生物学、蛋白工程、抗体/酶设计的研究人员。
  2. 需要把蛋白 embedding、结构预测接入自家分析流程的机器学习工程师或生信平台开发者。

- **怎么上手**：`pip install esm@git+https://github.com/Biohub/esm.git@main`

- **可以用在哪些场景**：
  1. 给内部蛋白序列库批量生成 embedding，用于功能聚类、相似性检索或下游分类模型训练。
  2. 对候选蛋白、抗原-抗体、蛋白复合物做结构预测，提前过滤实验优先级。
  3. 结合 SAE 和 Atlas，分析模型在不同蛋白家族上学到的可解释特征，用于功能注释或发现异常序列。

- **技术看点**：ESMC 是大规模蛋白语言模型，强调相对 ESM2 在长程结构理解上的扩展能力；ESMFold2 则把 ESMC 6B embedding 和扩散式结构预测架构结合起来。另一个很有辨识度的点是引入 SAE 去拆解隐藏层表征，把“模型知道了什么”尽量变得可解释，这对科研场景比单纯追指标更有价值。

- **同类对比**：README 明确提到 ESMFold2 在多项评测上可匹配或超过 AlphaFold3，并强调在单序列模式下有数量级的速度优势；同时 ESMC 被描述为相对 ESM2 的新一代扩展版本。

- **注意事项**：项目创建于 2024-06，更新很新，说明仍处于快速演进阶段；同时有 71 个 open issues，接口和依赖存在继续变动的可能。当前安装方式还是直接从 GitHub 安装，README 也写了“PyPI release is coming soon”，说明发布形态尚未完全稳定。
  另外，这类模型本身对显存、下载体积和 Hugging Face 权限有要求，本地跑大模型门槛不低；如果只是做推理验证，用 Biohub 平台 API 可能更省事。README 内容比较丰富，但更偏研究与能力展示，真正落地前最好先看 tutorials 和模型卡。

- **GitHub**：[Biohub/esm](https://github.com/Biohub/esm)
