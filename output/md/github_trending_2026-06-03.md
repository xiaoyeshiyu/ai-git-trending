### ✨ chopratejas/headroom (6754★)

> **一句话**：它把日志、工具输出、RAG 检索结果、代码文件和对话历史先压缩到原来的很小一部分，再送进大模型，让 AI 代理少花 60%～95% 的 token 也尽量给出同样答案。

- **它是什么**：Headroom 是一层加在 AI agent 和 LLM 之间的“上下文压缩层”。它既能作为 Python/TypeScript 库直接调用，也能以代理、命令行包装器、MCP 服务的方式接入 Claude Code、Codex、Cursor、Aider、Copilot 等工具。
  它不是简单截断文本，而是按内容类型走不同压缩路径：JSON 用 SmartCrusher，代码用 AST 感知的 CodeCompressor，普通文本可走自带模型 Kompress-base；同时用 CCR 保留原文，模型需要时还能按需取回。

- **能解决什么痛点**：
  - AI 编程或排障时，日志、搜索结果、代码片段一多就迅速吃满上下文窗口，导致调用成本高、响应慢，甚至关键内容被截断。
  - 同一个团队混用 Claude、Codex、Cursor 等多个 agent 时，上下文和“记忆”分散在不同工具里，重复喂给模型既贵又乱。

- **适合谁用**：
  - 高频使用 Claude Code、Codex、Cursor、Aider 等 AI 编程代理的开发者和团队。
  - 会把大量日志、检索片段、代码上下文送给 LLM 的平台工程、SRE、RAG/Agent 应用开发者。

- **怎么上手**：`pip install "headroom-ai[all]" && headroom wrap claude`

- **可以用在哪些场景**：
  - 让 AI 帮你排查线上事故时，先压缩成千上万行日志和告警上下文，再把关键异常送给模型分析。
  - 给代码助手做大仓库探索，把搜索命中、目录树、多个文件片段压缩后再喂给 Claude/Codex，减少超长上下文带来的费用和丢信息问题。
  - 在自建 RAG 或多代理系统里，把检索出的 chunk、历史对话和工具输出统一压缩，通过代理或 SDK 无侵入接入现有调用链。

- **技术看点**：项目的亮点在于“按内容路由”的压缩设计，而不是统一做摘要：结构化数据、代码、自然语言分别处理，更适合真实 agent 工作流。
  另一个值得关注的是可逆压缩（CCR）和 CacheAligner：前者让原文不丢失、可按需回取，后者尝试提高模型侧 KV cache 命中率，这对成本和时延都有现实意义。

- **近期动向与发展方向**：最近提交非常活跃，6 月初连续更新，说明项目仍在快速迭代。近期重点主要有三类：一是文档和 README 持续补强，二是围绕 Codex 的模型发现、日志和初始化行为做修复，三是对 docs 依赖和 LiteLLM 等组件做安全升级，说明作者在同时补可用性和工程稳定性。
  从提交者来看，不只有作者本人，也有多位社区贡献者参与 PR，社区活跃度不错。结合 README 内容，项目接下来大概率会继续强化多 agent 兼容、MCP 集成、跨 agent 记忆和评测体系。

- **同类对比**：README 没有明确列出直接竞品，但它明显不是只做“提示词压缩”或“对话摘要”的单点工具。相比依赖单一模型厂商原生上下文压缩能力的方案，Headroom 更强调跨代理、跨提供商、可逆压缩和本地优先运行。

- **注意事项**：项目创建于 2026-01-07，时间还很新，但已经有 6754 Star，增长很快；与此同时 Open Issues 有 130，说明关注度高，也意味着接口和行为仍可能快速变化。
  文档表面上比较完整，README 提供了安装、架构、基准和兼容矩阵，但近期仍在频繁修文档和修配置，说明文档体系还在打磨。若准备在线上接入代理模式、跨 agent 记忆或 `headroom learn`，建议先在测试环境验证压缩后准确率、回取链路和与现有工作流的兼容性。
  此外它依赖本地进程运行，README 也明确提到在受限沙箱环境里不一定适用；对只用单一模型厂商原生压缩能力、且没有跨工具协作需求的团队，收益未必最大。

### ✨ reconurge/flowsint (4559★)

> **一句话**：把域名、IP、邮箱、社交账号、网站等线索拖进同一张关系图里，自动跑各类 OSINT 富化任务，帮助安全分析员顺着图谱一步步还原目标之间的联系。

- **它是什么**：Flowsint 是一个面向网络安全分析和调查场景的开源 OSINT 图谱平台，核心体验不是“搜一条结果”，而是把调查对象变成图中的节点，再通过 enrichers 持续补全关联信息。它支持域名、IP、ASN、邮箱、手机号、社交账号、组织、加密钱包、网站等多类实体，并提供可视化图界面、自动扫描/富化、API 服务和实时事件流。README 还强调数据默认保存在本机，适合对隐私敏感的调查工作流。
- **能解决什么痛点**：一是调查时信息散落在 DNS、WHOIS、社交搜索、泄露库、网站抓取等不同工具里，手工切换和整理关系非常费劲；Flowsint 把这些结果统一落到图谱里，方便追链。二是做资产侦察或事件溯源时，经常要从一个域名继续扩到子域、IP、ASN、组织、网站链接等多跳关联，手工做很容易漏线索，这类图式富化流程正是它的强项。
- **适合谁用**：网络安全分析师、威胁情报研究员、OSINT 调查人员；也适合需要做内部数字资产梳理、风险暴露排查的安全团队。
- **怎么上手**：`git clone https://github.com/reconurge/flowsint.git && cd flowsint && make prod`
- **可以用在哪些场景**：1）做外部攻击面梳理：从公司域名扩展到子域、IP、ASN、网站与追踪脚本，整理暴露面。2）做钓鱼或诈骗调查：把邮箱、手机号、社交账号、网站、组织信息串成关系图，看是否指向同一主体。3）做链上调查或情报补充：从钱包地址继续追交易、NFT 持有情况，再结合组织或网站线索交叉验证。
- **技术看点**：项目采用清晰的模块化拆分：前端 app、FastAPI API、core、enrichers、types 分层独立，适合持续扩展新实体类型和新富化器。前端主打大规模图谱交互，后端则结合 PostgreSQL、Neo4j、Celery、Pydantic v2 等，明显是按“可扩展调查平台”而不是单点脚本来设计的。
- **近期动向与发展方向**：最近提交非常活跃，5 月到 6 月连续有合并和修复，说明项目仍在快速迭代。近期重点明显偏向稳定性和生产可用性，包括时区感知时间戳、社交富化输出编码、结果持久化、CORS 限制、非 root 容器运行、认证密钥加载、Celery 健康检查等，说明团队正从“功能能跑”推进到“部署更稳、更安全”。同时也有 collection enrichers、文档同步和技能脚本更新，表明功能扩展仍在持续，且已有外部贡献者参与。
- **同类对比**：暂无明显同类对标。
- **注意事项**：项目创建于 2025-01-31，虽然 star 增长很快，但 README 明确写了仍处于 early development，成熟度不能按老牌安全平台看待。当前有 40 个 open issues、贡献者 11 人，更新频率高，意味着活跃但也可能伴随接口和行为调整；如果要用于正式团队流程，建议先在测试环境验证升级兼容性。部署依赖 Docker 和 Make，整体门槛不算低；测试套件 README 也标注为 incomplete，说明质量保障体系还在补齐中。
