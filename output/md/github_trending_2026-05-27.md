### ✨ DigitalPlatDev/FreeDomain (167759★)

> **一句话**：它把“买域名”这件事变成了“直接领一个可用域名并挂到自己的 DNS 服务商上”，让个人和组织几乎零成本拥有公开可访问的网址。

- **它是什么**：这是一个提供免费域名注册服务的开源项目，核心不是本地软件，而是一套围绕免费域名分发、管理和文档说明的公开仓库。README 里明确给出了可申请的域名后缀，如 `.dpdns.org`、`.us.kg`、`.qzz.io`、`.xx.kg`、`.qd.je`，并支持把域名托管到 Cloudflare、FreeDNS by Afraid.org、Hostry 等 DNS 提供商上使用。
- **能解决什么痛点**：一是个人开发者、学生或小团队在做作品集、临时演示站、实验性项目时，不想为正式域名年费买单；二是很多人已经有自己熟悉的 DNS 服务商，但缺的是一个可公开访问、可绑定解析的域名入口，这个项目正好补上这一层。
- **适合谁用**：预算有限、想给个人网站或副项目挂独立网址的学生和独立开发者；需要批量给活动页、测试站、社区项目分配外部访问域名的小团队或非营利组织。
- **怎么上手**：直接访问控制台申请：https://dash.domain.digitalplat.org/
- **可以用在哪些场景**：给个人主页、博客、作品集绑定一个正式域名；给开源项目 Demo、黑客松作品、课程作业站点提供对外访问地址；给小型社区、公益组织或内部实验服务先挂免费域名验证想法。
- **技术看点**：项目强调“免费注册 + 自选 DNS 托管”的模式，用户不被绑定在单一解析平台上，实际部署灵活度比很多封闭式免费二级域名服务更高。README 还给出了教程、FAQ、滥用举报渠道和社区入口，说明它不是单纯放出域名后缀，而是在做一个持续运营的域名服务体系。
- **同类对比**：README 未直接列出竞品，但从定位看，它和常见“免费二级域名分发平台”相近；差异点在于明确支持接入 Cloudflare 等主流 DNS 服务商，而不是要求全部解析都留在平台内部。
- **注意事项**：这是 2024 年创建的较新项目，但更新到 2026-05-27，活跃度还在；Star 很高，但贡献者只有 2 人，说明核心维护比较集中。当前有 33 个 Open Issues，文档覆盖了教程、FAQ、社区和 abuse report，基础说明算完整，但 README 没提供 API、CLI 或自动化接入示例。它本质上依赖在线服务可用性，使用前应关注域名规则、滥用处理、后缀稳定性以及是否适合长期正式业务。

- **GitHub**：[DigitalPlatDev/FreeDomain](https://github.com/DigitalPlatDev/FreeDomain)

### ✨ jellyfin/jellyfin (52436★)

> **一句话**：把你自己的电影、剧集、音乐和照片集中放到一台服务器上，再通过网页、电视盒子、手机等设备随时播放和管理的开源媒体服务后端。

- **它是什么**：Jellyfin 是一个自由软件媒体系统的服务端与 API，核心能力是管理本地媒体库、提供流媒体播放、并把内容分发到多种客户端设备。它定位上直接对标 Emby、Plex 这类家庭媒体服务器，但强调完全开源、无付费墙、无高级功能分级。这个仓库主要是后端服务器本体，不包含 Web 客户端源码本身。
- **能解决什么痛点**：一是很多人手里有 NAS、家用服务器或硬盘里的大量影视和音乐文件，但文件夹混乱、封面和元数据分散，Jellyfin 可以统一整理、索引和播放。二是相比商业媒体服务器，用户常遇到远程播放、硬件转码、移动端使用等功能被订阅限制，Jellyfin 主打没有高级版/普通版之分。
- **适合谁用**：家庭影音爱好者、NAS 用户、想自建私人媒体中心的人；也适合需要自己掌控数据、不想依赖商业云媒体平台的开源用户和家庭实验室玩家。
- **怎么上手**：`dotnet run --project Jellyfin.Server --webdir /absolute/path/to/jellyfin-web/dist`
- **可以用在哪些场景**：在家里的 NAS 或小主机上搭建私人影视库，给电视、平板和手机统一看片；在局域网里给家庭成员共享音乐、照片和电视剧资源；在自托管环境中替代 Plex/Emby，避免账号绑定、订阅限制和闭源依赖。
- **技术看点**：后端基于 .NET 生态，强调跨平台运行；媒体处理依赖 ffmpeg，并提供 Swagger API 文档，说明它不只是“能播”，也适合和其他客户端或自动化系统集成。
- **同类对比**：README 明确提到它是 Emby 和 Plex 的替代方案；差异点在于完全自由软件、没有 premium license、没有隐藏付费功能，控制权更偏向用户自己。
- **注意事项**：这是个很成熟的大项目，2018 年创建、至今仍持续更新，社区规模大，但也有 677 个 open issues，说明功能面广、问题类型也多。上手不算一键傻瓜式：源码运行需要 .NET 10 SDK、额外准备 jellyfin-web 静态文件，还要安装 ffmpeg；如果只是想使用，建议优先走官方下载页和安装文档，而不是直接从源码起步。文档总体完整，覆盖安装、开发、测试和 Codespaces，但因为项目体量大、涉及客户端与转码链路，升级时仍需留意版本兼容和部署差异。

- **GitHub**：[jellyfin/jellyfin](https://github.com/jellyfin/jellyfin)

### ✨ st-tech/ppf-contact-solver (3566★)

> **一句话**：它把布料、实体和绳索之间复杂到会互相挤压缠绕的接触过程放到 GPU 上求解，让你能在 Blender、JupyterLab 或 Docker 里直接跑出高精度物理仿真结果。

- **它是什么**：这是 ZOZO 开源的接触求解器，面向物理仿真里的三类对象：shell（薄壳/布料）、solid（实体）和 rod（杆/绳）。项目不仅提供底层求解能力，还配了 Blender 插件、JupyterLab、Docker 镜像和 Windows 可执行包，目标是让用户不用自己搭一整套仿真环境也能直接跑案例。README 强调它不是实时引擎，而是偏离线计算，但在精度、稳定性和大规模接触数量上很激进。
- **能解决什么痛点**：一是做布料、绳索、柔性体仿真时，最容易遇到“物体互相穿透、卡住、拉伸变形失真”，这个项目主打 penetration-free contact 和严格应变上界，专门解决这类结果不可信的问题。二是很多高质量接触仿真一到大规模场景就跑不动，这个项目明确给出了超过 180M contacts 的极端案例，适合需要把复杂接触场景真正算下来的用户。
- **适合谁用**：一类是做物理动画、数字服装、几何处理、计算机图形学研究的开发者和研究人员。另一类是需要借助 Blender 或云端 GPU 跑离线仿真的技术美术、仿真工程师，以及希望把求解器接到 Python 工作流里的团队。
- **怎么上手**：README 给出的最省事方式是 Windows 下双击 `start.bat`，或在 Linux/Windows 上直接运行官方 Docker 镜像；文档未提供单行命令级的统一快速上手示例。
- **可以用在哪些场景**：一是在 Blender 里远程提交服装或柔性物体仿真任务，本地取回结果，适合 macOS 端做内容制作。二是用 JupyterLab 跑论文复现、参数实验和批量案例分析，适合研究和算法验证。三是在云 GPU 上部署大规模接触仿真，例如编织结构、扭转绳索、多物体挤压等高接触密度场景。
- **技术看点**：核心卖点是 GPU 上同时跑接触与弹性求解，并且全部使用单精度，强调缓存效率和可扩展性。方法上基于 FEM，并提供 symbolic force jacobians；同时项目背后有 TOG 论文《A Cubic Barrier with Elasticity-Inclusive Dynamic Stiffness》作为技术来源，不只是工程封装。
- **同类对比**：暂无明显同类对标。README 更强调自身特点：高接触规模、严格防穿透、Windows 原生可执行、Docker 封装、Blender 插件和 MCP 支持，而不是直接与某个现成仿真框架做横向比较。
- **注意事项**：首先，硬件门槛不低，需要支持 CUDA 12.8+ 的 NVIDIA GPU，且仅支持 x86，不支持 arm64。其次，项目创建于 2024-11，仍属较新的高性能仿真项目，虽然更新很勤、CI 和文档都比较完整，但 2025-10 有过一次“大规模重构并包含 Python API breaking changes”，说明接口稳定性还在演进中。再者，当前贡献者只有 2 人，Open Issues 为 12，维护活跃但 bus factor 偏低；如果你要在生产流程里深度集成，建议优先锁定版本或直接使用文档提到的参考分支。
