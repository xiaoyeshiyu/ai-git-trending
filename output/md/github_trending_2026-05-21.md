### ✨ can1357/oh-my-pi (5525★)

> **一句话**：把终端里的 AI 编程助手直接接上 IDE 能力、调试器、浏览器和代码执行环境，让它不只会“改几行字”，而是真能在仓库里读、查、改、跑、调、审。

- **它是什么**：这是一个面向终端的 AI Coding Agent，核心不只是聊天补全，而是给模型接入完整开发工具链：文件读写、搜索、LSP 重命名/引用、DAS 调试、Python 与 Bun 执行、网页与 PDF 阅读、子代理并行协作等。README 里强调它做了大量“工具层”优化，比如基于内容哈希的编辑锚点、面向不同模型调优的提示词、原生集成搜索与文件系统接口，目标是让模型第一次就把改动落到正确位置。
- **能解决什么痛点**：一是很多 AI 编程工具在复杂重构时容易把文件改坏，比如重命名涉及 barrel file、别名导入、跨文件引用时常常漏改；这个项目把 LSP 直接接入写操作，能按 IDE 语义做 rename/ref。二是排查运行时问题时，普通 AI 往往只能建议你“加日志试试”，而它能直接驱动 lldb、dlv、debugpy 这类调试器，在真实进程里停住、看栈、查变量。
- **适合谁用**：经常在本地终端里处理多语言仓库、需要 AI 深度参与重构与排障的全栈工程师；以及希望把 AI 用在代码审查、调试、批量修改、跨文件分析上的工具链开发者、基础设施工程师。
- **怎么上手**：`bun install -g @oh-my-pi/pi-coding-agent`
- **可以用在哪些场景**：1）在大型 TypeScript/Go/Rust 项目里做跨文件重命名、接口迁移、模块拆分，减少“改一处漏三处”的问题；2）排查 C/C++ 崩溃、Go 服务卡死、Python 进程阻塞时，让代理直接连调试器定位现场；3）让代理读取 GitHub PR、Issue、Arxiv PDF、网页文档，再结合本地仓库完成修复、评审或知识整理。
- **技术看点**：项目前端表面是 TypeScript，但 README 明确提到有约 2.7 万行 Rust core，说明其高频工具链能力走了原生实现路线，而不是大量 shell out 外部命令。另一个关键设计是 Hashline 内容哈希锚点编辑，避免模型反复重写上下文、字符串匹配失败和无效 diff 重试，对稳定性和 token 成本都有直接价值。
- **同类对比**：README 明确提到它源自 Mario Zechner 的 Pi，并在其基础上补全了代码执行、LSP、DAP、子代理、规则注入、GitHub/PR 文件系统化访问等能力。它也多次对比“多数 agent/harness 只给一个 Python sandbox”或依赖外部 grep/bash/rg 的方案，差异点在于工具面更完整、原生集成更深。

### ✨ truelockmc/streambert (3232★)

> **一句话**：把全球电影、剧集和动漫的在线播放、下载、字幕管理整合进一个跨平台桌面播放器里，打开应用就能搜、看、下，而且默认无广告、无追踪。

- **能解决什么痛点**：一是在浏览器里找片源、切换站点、被广告和弹窗打断的观影流程，这个应用把搜索、播放、字幕和下载收进一个本地客户端里。二是遇到 m3u8 流媒体想离线观看时，普通用户往往还要借助浏览器插件和命令行工具，这个项目把链路打通到了应用内。

### ✨ zakirullin/files.md (2305★)

> **一句话**：把本地 Markdown 文件变成一个可离线使用、像聊天一样随手记笔记和任务的私有思考空间，所有内容都直接落在你自己的文件夹里。

- **它是什么**：这是一个围绕 `.md` 文件构建的本地优先笔记应用，核心思路不是做复杂的知识管理系统，而是让你用浏览器就能打开自己的文件夹，直接写笔记、记日记、收任务、建清单。它支持聊天式快速记录、文件间链接、离线使用，以及基于本地文件或自托管服务的同步；数据默认不发到服务器，文件结构也尽量保持简单明了。
- **能解决什么痛点**：一是很多人想用 Markdown 管理笔记，但又不想把内容锁进云服务或重型笔记软件里，这个项目直接以本地文件为中心，迁移和备份都简单。二是灵感、待办、购物清单这类碎片信息经常来得很快，传统笔记工具打开和归类步骤多，这里可以直接像发消息一样先记下来，再决定存到笔记、任务或日记里。
- **适合谁用**：适合偏爱纯文本、本地文件和可控数据的开发者、写作者、研究者；也适合已经在用 Markdown 做知识管理，但觉得 Obsidian 一类工具过重、插件过多的人。
- **怎么上手**：最简单方式是直接打开 https://app.files.md ，在浏览器中点击“Install files.md”，然后选择一个本地文件夹用于持久化保存。
- **可以用在哪些场景**：一是开发者把技术笔记按主题放进 `brain/`、`dev/` 等目录，边写边互相链接，整理长期知识网络。二是日常工作中把临时待办、跟进事项、采购清单先丢进聊天入口，稍后再归档到 `Later.md` 或固定 checklist。三是做长期 journaling，把每天的记录自动沉淀到 `journal/YYYY.MM Month.md`，保持纯文本可检索、可同步。
- **技术看点**：项目强调 local-first 和 plain files，前端可直接通过浏览器使用，本地数据不出设备；同步层也尽量克制，要么直接用 iCloud/Dropbox/Google Drive 这类文件夹同步，要么只跑一个 Go 二进制作为自托管同步服务。另一个很鲜明的点是代码和结构刻意保持简单，README 明确面向“一个人或 LLM 能吃透整个项目”的扩展方式。
- **同类对比**：README 明确提到了 Obsidian/Second Brain 路线，并有意做反方向取舍：少功能、少结构、少插件，弱化“搭系统”的兴奋感，强调把注意力放回思考本身，而不是沉迷模板、图谱和工作流。
- **注意事项**：项目从 2023 年开源、作者称已打磨 5 年，近期仍在更新，8 个 open issue、贡献者 7 人，活跃度和维护状态看起来不错，但线上应用仍标注为 Beta。上手门槛不高，浏览器即可体验，文档也比较完整；不过它的理念很强，若你需要数据库级全文检索、复杂插件生态、团队协作权限或高度自动化的 PKM 工作流，可能会觉得功能过于克制。另外 README 提到偶尔需要强制刷新获取更新，说明 Web 端缓存和版本切换体验还不算完全无感。
