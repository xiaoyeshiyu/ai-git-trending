✨ janestreet/magic-trace (5912★)

一句话：它能盯住一个正在运行的 Linux 进程，把触发点之前大约 10ms 内的每一次函数调用按时间线展开，让你直接看到程序在卡顿、崩溃前到底做了什么。

  • 它是什么：magic-trace 是 Jane Street 开源的高分辨率程序追踪器，核心能力不是“抽样看个大概”，而是基于 Intel Processor Trace 抓取某个时间点之前的完整控制流。它可以像 perf 一样附着到进程上，在你按下 Ctrl+C 或命中特定函数时截取快照，然后用交互式时间线回放调用栈，分辨率可到约 40ns。它不要求改应用代码，适合直接分析线上或本地已有进程。
  • 能解决什么痛点：一是排查“偶发慢请求”时，传统 profile 往往只看到平均热点，看不到某次异常请求在最后几毫秒里具体经历了什么；二是程序崩溃后通常只有最终栈，而它能保留崩溃前的一小段执行历史，帮助定位“是怎么走到这里的”。
  • 适合谁用：做 Linux 性能分析、线上疑难排障的后端工程师和 SRE；需要看清函数级执行细节的底层/系统工程师，尤其是经常用 perf、调 C/C++/原生运行时的开发者。
  • 怎么上手：安装发布二进制后，直接附着进程：magic-trace attach -pid $(pidof demo)
  • 可以用在哪些场景：排查生产环境中少量超时请求，观察它们在进入业务函数前后是否卡在页错误、动态链接或调度切换上；分析某个编译器 pass、GC 结束后或异步调度周期过长时，前几毫秒到底执行了哪些函数；调试“程序突然崩了但栈信息不够”的场景，补上崩溃前的调用时间线。
  • 技术看点：它不是常见的采样 profiler，而是借助 Intel PT 记录控制流，再通过 perf 驱动和解释底层追踪数据，因此能在很短时间窗内还原“每个函数调用”而不是少量样本点。README 给出的开销大约在 2% 到 10%，这是它在可观测性和侵入性之间比较有吸引力的平衡点。
  • 同类对比：README 明确对标 perf。区别在于 perf 常见用法是周期性采样，适合看整体热点；magic-trace 更擅长围绕某个触发时刻回看完整调用片段，尤其适合分析几十纳秒到几微秒级别、采样工具容易漏掉的细节。

  • 注意事项：硬件和环境限制很强：仅支持 Linux、仅限 Intel，且通常要求 Skylake 及之后平台，虚拟机大多不支持，这会直接决定你能不能用。项目自 2022 年以来持续维护，24 位贡献者、55 个 open issues，最近仍有更新，成熟度不错；README 和 Wiki 写得比较细，入门路径清晰。但它依赖底层追踪能力和符号信息，真正在线上用好仍需要一定 Linux/perf 背景，不算零门槛工具。
